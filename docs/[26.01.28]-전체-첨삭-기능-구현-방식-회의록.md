## 🗣️ 논의 내용
### 1. 첨삭 데이터 저장 방식
**블록 단위 인덱스 방식 (채택 X)**
- 드래그가 큰 단위로 일어나는 경우 DB 연산이 과도하게 발생
- 성능 이슈로 인해 기각

**스크립트 자체 저장 방식 (채택)**
- 예시: `%c1%열정적인%c1% 사람입니다. 그리고 %c2%성실함%c2%을 중요시합니다.`
- c1, c2 등의 태그로 첨삭 위치 표시
- 각 태그에 댓글 내용 매핑

### 2. 첨삭 시나리오 및 제약사항
- 첨삭자는 한 명으로 제한
- 중첩 첨삭은 지원하지 않음
- 라이브러리 우선순위: Yjs > Slate.js > ~~LexoRank(제외)~~

### 3. Frontend 추가 검토 사항
- 블록 간 링크드리스트 구조 (실현 가능성 낮음)
- Slate.js 라이브러리 상세 조사 필요
- 한글 입력 지원 기능 추가 검토

## ✅ 결론 및 결정 사항
**유저 시나리오 확정**
1. BE가 FE에게 스크립트 형태로 문서 데이터 전달
2. 첨삭자가 첨삭 시 FE는 BE에게 변경점(delta)만 전송
3. BE에서 데이터 갱신 후 다시 FE로 전달

**채택된 방식**
- 스크립트 자체에 첨삭 태그를 삽입하는 방식으로 구현
- 한 명의 첨삭자만 지원, 중첩 첨삭 미지원
