# 🤔 [고민한 주제] 프로젝트 백로그, 스프린트 룰

## 📝 고민 배경
- 왜 이 고민을 하게 되었는가? 팀원마다 백로그와 스프린트를 정의하는 개념이 조금씩 달랐다. 개발 일정 관리를 효율적으로 하기 위해서는 용어와 관리 방식을 통일할 필요가 있다고 판단했다.
- 전제 조건: 팀원 모두가 직관적으로 이해할 수 있어야 하며 프론트엔드와 백엔드 간의 작업 분리가 명확해야 한다.
- 제약 사항: 복잡한 관리 툴(Jira의 복잡한 워크플로우 등)을 도입하기보다는 현재 사용 중인 도구(Github Project)에서 합의된 규칙으로 해결하기로 했다.

## 🔄 고려한 선택지
### 선택지 1: 배포 단계별 이터레이션 & 기능 중심 계층 구조
- 개념
  - 이터레이션: 1차 배포, 2차 배포(성능 개선), 최종 배포 등 굵직한 마일스톤 기준
  - 스프린트: 큰 기능 단위
  - 백로그: 세부 기능 단위
- 장점
  - 개발 목표(배포)와 구현해야 할 기능이 직관적으로 매칭된다.
  - **무엇을 개발하는가** 가 명확하게 보인다.
- 단점
  - 사용자 스토리보다는 기능 명세서에 가까워 사용자 관점이 다소 약해질 수 있다.

### 선택지 2: Jira 표준 방식 (Epic-Story-Task)
- 개념
  - Epic(큰 줄기) -> Story(사용자 시나리오) -> Task(작업) 형식
- 장점
  - 사용자 시나리오 중심이라 **누가, 왜 이 기능을 쓰는가** 를 이해하기 좋다.
  - Jira에서 많이 쓰이는 방식이다.
- 단점
  - 시나리오 위주로 작성되다 보니 개발자가 당장 구현해야 할 기능적 실체가 뚜렷하게 보이지 않는다.
  - 기술적인 세부 구현 사항을 관리하기에 모호한 부분이 있다.
  - 처음부터 완벽한 체화가 힘들다.

### 선택지 3: 시나리오 Parent & FE/BE Sub-task 분리
- 개념
  - Parent: 사용자 시나리오 기술
  - Child: 하위에 FE 개발 할 일, BE 개발 할 일을 각각 작성
- 장점
  - 프론트엔드와 백엔드의 할 일이 명확히 구분된다.
- 단점
  - Parent(시나리오)와 FE Task의 내용이 일치하는 경우가 생겨서 Parent와 FE의 경계가 모호해진다.

## ✅ 최종 선택
- 선택한 방안: 선택지 1 (배포 단계별 이터레이션 & 기능 중심 스프린트/백로그)
- 선택 이유
  - 팀의 현재 상황에서는 시나리오의 나열보다 어떤 기능을 구현할지 뚜렷하게 보이는 것이 더 중요하다고 판단했다.
  - 단계별 목표를 이터레이션으로 설정하고 일정 관리하는 것이 용이하다고 판단했다.

## 📉 아쉬운 점
- 포기한 부분
  - 선택지 2번의 사용자 스토리 중심 서술 방식이 주는 사용자 입장의 맥락을 포기했다. 백로그 작성 시 기능 구현 위주로 흐를 수 있다.
- 리스크
  - 큰 기능의 정의를 잘못하면 하나의 스프린트가 너무 커지고 백로그가 많아질 수 있고 백로그가 어떤 시나리오에서 사용되는지 놓칠 수 있다.

## 🔮 이후 개선 방향
- 추후 다시 고려할 사항
  - 프로젝트가 고도화되어 기능 명세만으로 사용자 니즈 파악이 어려워질 경우 사용자 시나리오를 백로그에 요약을 한 줄 추가하는 방식을 고려해 볼 수 있다.
  - 스프린트 하나가 너무 커지면 단위를 더 잘게 쪼개는 기준을 마련해야 한다.
