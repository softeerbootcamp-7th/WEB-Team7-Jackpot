# 🤔 [고민한 주제]

## 📝 고민 배경
- 왜 이 고민을 하게 되었는가? 기존에 React Router에서 페이지를 이동시킬 때 주로 useNavigate를 사용하였지만 팀원에게 Link를 사용해보는 것이 어떻냐는 코드 리뷰를 받았다.
- 전제 조건: React Router 라이브러리를 사용하는 SPA 환경
- 제약 사항: 검색 엔진 최적화(SEO)와 웹 접근성을 고려해야 하며 UX를 저해하지 않아야 한다.

## 🔄 고려한 선택지
### 선택지 1 useNavigate (명령형 방식)
- 개념: react-router에서 제공하는 훅으로 특정 이벤트(클릭, 데이터 전송 완료 등)가 발생했을 때 코드로 URL을 변경하는 함수를 반환한다.
- 장점
  - 조건부 이동: 로그인 성공 여부나 폼 제출 결과에 따라 이동할지 말지 결정할 수 있다.
  - 추가 로직 실행: 이동 전에 이벤트를 전송하거나 상태를 변경하는 등 함수 내부에서 다양한 작업을 수행 가능하다.
  - History 조작: `{ replace: true }` 옵션으로 뒤로 가기를 막는 등 히스토리 스택 제어가 용이하다.
- 단점
  - SEO 불리: 크롤러가 자바스크립트를 실행하지 못하는 경우 링크를 따라가지 못할 수 있다.
  - UX 제한: 사용자가 `Ctrl + 클릭`이나 마우스 휠 클릭으로 새 탭에서 열기를 할 수 없다.
  - 접근성 낮음: 스킌 리더가 단순 버튼이나 `div`로 인식하여 어디로 이동하는지 미리 알기 어렵다.

### 선택지 2 Link (선언형 방식)
- 개념: HTML의 `<a>` 태그를 기반으로 하지만 브라우저의 새로고침을 막고 CSR을 수행하는 컴포넌트이다.
- 장점
  - 웹 표준 준수: 실제 DOM에는 `<a>` 태그로 렌더링되므로 시맨틱하다.
  - 우수한 UX: 사용자가 새 탭에서 열기, 링크 주소 복사 등 브라우저 고유 기능을 사용할 수 있다.
  - SEO 친화적: 크롤러가 `href` 속성을 보고 사이트 구조를 파악하고 페이지를 수집하기 용이하다.
  - 접근성 우수: 스크린 리더가 링크로 인식한다.
- 단점
  - 스타일링: `<a>` 태그의 기본 스타일을 제거하기 위한 추가 CSS 작업이 필요할 수 있다.
  - 조건부 로직 불가: 클릭 즉시 이동하므로 이동 전 복잡한 비즈니스 로직을 태우기에는 부적합하다. (단순 이동에 특화)

### 선택지 3 NavLink (스타일링 특화 방식)
- 개념: `Link`의 특수 버전으로 현재 URL과 링크의 경로가 일치할 때(활성화 상태) 자동으로 특정 스타일이나 클래스를 적용해준다.
- 장점
  - 활성화 상태 표시: 네비게이션 바 등에서 현재 위치를 시각적으로 표현하기 쉽다. (`isActive` 프로퍼티 제공)
  - `Link`의 모든 장점을 그대로 가진다.
- 단점
  - 단순 이동 버튼 등 현재 상태를 표시할 필요가 없는 곳에 쓸 필요가 없다.

## ⚖️ 비교 정리
항목 | 선택지 1 (useNavigate) | 선택지 2 (Link) | 선택지 3 (NavLink)
-- | -- | -- | --
주 사용 목적 | 로직 처리 후 이동 (로그인, 결제 등) | 단순 페이지 이동 (메뉴, 배너 등) | 현재 위치 표시가 필요한 메뉴
SEO 점수 | 낮음 (JS 의존) | 높음 (Semantic Tag) | 높음 (Semantic Tag)
접근성 | 낮음 (role 부여 필요) | 높음 (Native <a>) | 높음 (Native <a>)
새 탭 열기 | 불가능 | 가능 | 가능
구현 난이도 | 쉬움 (함수 호출) | 쉬움 (컴포넌트 사용) | 보통 (스타일링 로직 추가)

## ✅ 최종 선택
- 선택한 방안: 단순 이동 버튼과 메뉴에는 `Link` or `NavLink`를 사용하고 `useNavigate`는 비즈니스 로직이 필요한 경우에만 제한적으로 사용한다.
- 선택 이유
  - 팀원의 피드백대로 웹 표준과 접근성을 지키는 것이 장기적으로 서비스 품질에 중요하다고 판단했다.
  - 사용자의 UX를 저해하지 않기 위한 개발을 하기로 했다.
  - SEO를 위해 시맨틱 태그를 사용하기로 했다.
