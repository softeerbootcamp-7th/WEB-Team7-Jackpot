## 🗣️ 논의 내용

### 1. 로딩 상태 UI 구현 방식
**현황**
- 서비스에서 사용하는 로딩 종류: Dots 로딩, 로딩 스피너 2가지

**논의된 구현 방식**
- 방법 1: HTML + CSS로 직접 구현
- 방법 2: 로딩 컴포넌트를 제공하는 라이브러리 사용

**결정**
- 2가지 로딩 타입의 구현 통일성을 위해 두 방식 모두 조사 후 결정
- 직접 구현 시, 공통 컴포넌트화 진행

### 2. 자기소개서 작성 페이지 디자인 통일
**배경**
- 현재 3가지 자기소개서 뷰의 디자인이 상이함

**결정 사항**
- 자기소개서 작성 페이지의 UI로 통일
- 우측에 첨삭 댓글 확인 탭 배치
- 첨삭 링크가 활성화된 자기소개서의 경우, 우측 탭이 함께 표시되는 형식 적용

**라이브러리 페이지 변경 사항**
- 기존 토글 리스트 형태 → 페이지네이션 방식으로 변경
- 수정하기 및 문항 스크랩 버튼 위치를 첨삭 페이지의 버튼 위치로 이동

### 3. 백로그 마일스톤 재정리
**문제점**
- 기존의 페이지 단위 마일스톤 분류로 인해 정리가 애매한 이슈 발생

**해결 방안**
- 공통 컴포넌트 마일스톤 신규 생성 후 재정리 진행

### 4. 자동 저장 API 요청 시점
**결정**
- 1분마다 자동 저장
- 첨삭 댓글 등록 시마다 자동 저장

## ✅ 결론 및 결정 사항
- 로딩 UI는 Dots 로딩과 스피너 2종류를 통일된 방식으로 구현 (직접 구현 vs 라이브러리 조사 후 결정)
- 자기소개서 관련 모든 뷰를 작성 페이지 UI 기준으로 통일, 우측 첨삭 탭 적용
- 라이브러리 페이지는 페이지네이션 방식으로 변경
- 공통 컴포넌트 마일스톤 생성 및 백로그 재정리
- 자동 저장은 1분 주기 + 첨삭 댓글 등록 시 트리거
